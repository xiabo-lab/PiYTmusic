## Ответы на ключевые вопросы

### 1. Риск бана / подозрительной активности со стороны Google

Как устроено сейчас (и это удачно):

- **API-запросы (браузинг, поиск, библиотека, лайки)** идут через InnerTube с куками
  пользователя + SAPISIDHASH — по форме это неотличимо от официального веб-клиента
  (`clientName: WEB_REMIX`, тот же UA, те же заголовки, visitorData echo). Это ровно то,
  что делает сам music.youtube.com.
- **Стриминг аудио НЕ привязан к аккаунту**: yt-dlp вызывается анонимно, без кук
  (комментарий в `lib.rs` это фиксирует осознанно — авторизованный yt-dlp Google
  как раз режет). Значит «качающую» активность нельзя атрибутировать аккаунту
  пользователя → риск бана аккаунта минимален by design.

Реалистичные риски:

| Риск | Вероятность | Последствие |
|---|---|---|
| Бан Google-аккаунта за использование | Очень низкая — прецедентов с третьесторонними плеерами (ytmdesktop, FreeTube и т.п.) не известно | — |
| Поломка стриминга (YouTube меняет протокол: PO tokens, SABR) | Высокая, регулярная | Играбельность падает до выхода новой версии yt-dlp |
| IP rate-limit / временные 403 при активном прослушивании | Низкая-средняя | Временные ошибки воспроизведения |
| Формальное нарушение ToS YouTube | Да, формально нарушает (третьесторонний клиент, обход рекламы) | Риск для проекта (DMCA/takedown репозитория), не для пользователей |

Выводы для релиза:
- В README и About — дисклеймер: *«Unofficial client. Not affiliated with Google/YouTube.
  Use at your own risk»*.
- **yt-dlp должен быть обновляемым** без пересборки приложения (см. вопрос 2) — это
  главная страховка от поломок.
- Не добавлять никакой «агрессивной» активности (массовые лайки, скрейпинг) — текущий
  профиль запросов повторяет обычную сессию.

### 2. Работоспособность у других пользователей / привязка к машине

Найдена **одна жёсткая привязка — блокер релиза**:

- `lib.rs` вызывает `Command::new("yt-dlp")` → yt-dlp берётся из **PATH**. У пользователей
  его нет → воспроизведение не заработает вообще.
  **Решение:** скачивать `yt-dlp.exe` при первом запуске (+ команда self-update) в
  AppData, либо бандлить как Tauri sidecar. Рекомендация — **скачивание + автообновление
  yt-dlp** (официальные релизы на GitHub), потому что yt-dlp чинится чаще, чем мы будем
  релизиться, и бандл в инсталляторе мгновенно устаревает.
- ffmpeg не требуется (качаем один аудиоформат без конвертации), но проверить на чистой
  VM обязательно.

Что проверено и привязки НЕ имеет:
- Куки — только собственный логин пользователя (DPAPI-шифрование, AppData).
- visitorData / Musixmatch-токен — получаются в рантайме, ничего не захардкожено.
- Никаких личных API-ключей в коде нет.

Прочие факторы совместимости:
- **WebView2** — на Win10/11 почти всегда есть; в bundle-конфиге включить
  `webviewInstallMode: downloadBootstrapper` на всякий случай.
- **Только Windows**: DPAPI, SMTC, windows-sys — кроссплатформенности нет. Первый релиз
  честно объявить Windows-only.
- **SmartScreen**: неподписанный installer даст «Windows protected your PC». Варианты:
  (а) релизить без подписи и задокументировать (норма для OSS), (б) Azure Trusted
  Signing (~$10/мес), (в) OV-сертификат (~$100+/год). Для старта — (а).
- Возможны false-positive антивирусов на yt-dlp — задокументировать в FAQ.

### 3. Логин и определение плана

Как работает:
1. **Логин** (`start_login` в lib.rs): открывается отдельное WebView-окно с настоящей
   страницей accounts.google.com (свежий профиль на каждую попытку), после успешного
   входа куки забираются из webview, шифруются DPAPI и складываются в
   `accounts/<id>/cookies.enc`. Пароль приложение не видит — логин полностью на стороне
   Google. Это безопасная схема, менять не надо.
2. **Premium-детект** (`fetchPremiumStatus`): эвристика по `account/account_menu` —
   есть апселл «Get Premium» → free; есть «Manage membership» → premium; ничего не
   распознали → fallback в premium + ручной override в настройках.

Что реально гейтится по плану (see `premium.ts`, stream-server):
- Premium → постоянный дисковый кеш треков + prefetch.
- Free → эфемерный кеш (стирается при каждом запуске), без prefetch.

**Важная честная оговорка:** полного паритета с ограничениями бесплатного YT Music
(реклама, ограничение фоновой игры) в приложении нет и технически не будет — стримы
анонимные, Google сам не присылает рекламу в этот канал. То, что мы контролируем и уже
делаем правильно: free-пользователь **не накапливает офлайн-библиотеку** (это
Premium-фича «загрузки»). Перед релизом:
- Прогнать детект на free-аккаунте и premium-аккаунте, на en и ru локалях.
- Убедиться, что у нового пользователя без логина всё работает в анонимном режиме
  (домашняя, поиск, воспроизведение) — это будет самый частый первый запуск.

### 4. Подключение GitHub (пошагово)

1. Аудит перед публикацией: `.gitignore` (dist, node_modules, target уже игнорятся —
   проверить), в истории git не должно быть кук/токенов (их там нет — куки живут в
   AppData), убрать личные упоминания из `Cargo.toml` (`authors = ["you"]`).
2. **LICENSE** — обязательный файл. Рекомендация: MIT (максимально простая) или GPL-3.0
   (если хотим, чтобы форки оставались открытыми). Без файла лицензии код формально
   «all rights reserved» и форкать/контрибьютить нельзя.
3. **Логотип/название**: `yt-music-logo-png.png` и иконки на основе логотипа YT Music —
   товарный знак Google, для публичного репозитория это прямой повод для takedown.
   Нужна своя иконка. Название `ytm-native` — приемлемо (аббревиатура), но не
   использовать «YouTube Music» в названии продукта.
4. Создание репозитория (через сайт github.com → New repository, или `gh repo create
   ytm-native --public --source . --push`). Описание, топики (tauri, youtube-music,
   desktop), скриншоты в README.
5. **GitHub Actions**: workflow на `tauri-action` — собирает NSIS-инсталлятор на тег
   `v*`, публикует draft-release с артефактами. Секреты подписи updater'а (см. п. 5) —
   в repo secrets.
6. Шаблоны issues (bug report / feature request), включить Discussions.

### 5. Автообновления внутри приложения

Стандартный путь Tauri 2 — **tauri-plugin-updater + GitHub Releases**:

1. Добавить `tauri-plugin-updater` (Rust) + `@tauri-apps/plugin-updater` (JS) и
   `tauri-plugin-process` (для relaunch).
2. Сгенерировать ключи подписи (`pnpm tauri signer generate`) — приватный в GitHub
   Secrets, публичный в `tauri.conf.json`.
3. В конфиге: `createUpdaterArtifacts: true`, endpoint —
   `https://github.com/<user>/ytm-native/releases/latest/download/latest.json`
   (tauri-action сам генерирует latest.json и .sig).
4. UI: оживить пункт «Updates» в top-bar — проверка при старте (тихая) + ручная
   проверка; тост «доступна версия X» → скачивание с прогрессом → «Restart to update».
5. Поток пользователя: обновление прилетает само, на GitHub ходить не нужно. ✔
6. Тест: собрать v0.1.0, поставить, выпустить v0.1.1, убедиться что подхватилось.

Отдельно: **автообновление yt-dlp** (не приложения) — проверять версию раз в N дней и
качать новый бинарь с github.com/yt-dlp/yt-dlp/releases. Это чинит стриминг без релиза
приложения.

### 7. Report Issue / фидбек с голосованием

Сейчас диалог-заглушка (форма никуда не шлёт). Варианты:

| Вариант | Голосование | Цена | Инфраструктура |
|---|---|---|---|
| **GitHub Issues + Discussions** | 👍-реакции, сортировка по ним | 0 | нет |
| Canny (как у Raycast-подобных) | да, полноценное | free-tier очень куцый, дальше $79+/мес | нет |
| Fider (self-hosted, OSS) | да | хостинг ~$5/мес | свой сервер |
| Featurebase / Sleekplan / Nolt | да | free-tier ограничен | нет |

**Рекомендация для старта: GitHub Issues + Discussions.** Ноль затрат, пользователи
голосуют 👍, разработчик отвечает, всё живёт рядом с кодом. Кнопку «Report Issue»
переделать: форма собирает title/body + автоматически подставляет версию приложения,
ОС, версию yt-dlp → открывает в браузере предзаполненный
`github.com/<user>/ytm-native/issues/new?title=...&body=...`. Не требует ни бэкенда,
ни OAuth. Минус — нужен GitHub-аккаунт у репортера; если это станет проблемой, позже
добавить Fider/Canny, кнопка просто поменяет URL.

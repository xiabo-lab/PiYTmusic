# План выполнения review (батчами, экономно по лимитам)

## Почему прошлый прогон сжёг лимиты и как этого избежать

Прошлый workflow за раз поднял ~98 агентов: 9 доменных ревьюеров **плюс** верификация, где каждая high-находка проверялась через 3 «линзы» (×3 агента), а каждая medium — ещё по агенту. Итог — лавина.

**Правила нового режима:**
1. **Ревью не повторяем** — 9 ревьюеров уже отработали, 72 находки лежат ниже в этом файле. Это готовый результат.
2. **1 агент на находку**, одна «refute»-линза вместо трёх. Это главный рычаг: убирает ×3.
3. **Жёсткий кап ≤ 8 агентов на один запуск workflow.** Каждый шаг ниже уже уложен в этот лимит.
4. **Фиксы делаю сам** (Edit/Write, 0 агентов веером) — агенты только для независимой верификации.
5. **Делаем по 2–3 шага за сессию.** Пик расхода за сессию — ~16–24 агента вместо 98.

# 📋 Финальный отчёт

## Вердикт

**ytm-native — сильный, продуманный проект существенно выше среднего уровня хобби-кода, но без единой страховки (git/линтер/тесты/CI) и с тремя системными слабостями, которые уже начали ломать поведение.** Архитектура зрелая: дисциплинированная слоистость (сырой InnerTube + парсеры → Zustand-сторы → UI), продуманный мост между окнами, повсеместные комментарии «почему», осознанные защиты от гонок. Проблемы кластеризуются не в «плохом коде», а в нескольких корневых причинах — их устранение локально и окупается многократно.

Обе статические проверки зелёные (`tsc --noEmit`, `cargo check`). Кодовая база ~17k строк (TS + Rust).

## Сильные стороны (не переписывать)

- **Границы файлов/процессов в Rust** сделаны аккуратно: yt-dlp через массив аргументов без shell (нет command injection), санитизация video_id/имён (нет path traversal), DPAPI-шифрование кук на Windows, magic-byte MIME-sniffing, floor 32 КБ против storyboard-заглушек.
- **Защиты от гонок там, где их обычно забывают**: resolve-token при резолве стрима, StrictMode-safe отписка Tauri-listener'ов, `partialize`, который никогда не автозапускает воспроизведение при старте, версионированные миграции сторов.
- **Парсинг InnerTube защищённый**: optional chaining, tree-walk с WeakSet-guard от циклов, token-echo и 200-page cap в пагинации, отличные комментарии о причудах ответов YT.
- **Thumbnail fallback-цепочка** (iTunes → disk-cache → крупнейший вариант, с демоцией по onError и отменой) — реально тщательная.

## Корневые причины (чинить здесь — закроет много находок сразу)

1. **Синхронизация состояния между окнами** — самая крупная архитектурная слабость. Плавающее окно — отдельный JS-контекст, делит ключи localStorage с главным по принципу «последний победил», а событие закрытия не различает намерение пользователя и программное закрытие. → high #7, #8; medium #5, #6, #7, #22; low layout.ts. **Фикс-направление:** плавающее окно без persist-кэша; полный набор действий в мосте; дискриминатор user-vs-programmatic close.
2. **Дрейф query-ключей** — ad-hoc строковые ключи разбросаны по 15+ файлам и уже разошлись (мёртвые инвалидации, мёртвый persist-whitelist, дубли кэша liked). → high #9; medium #13, #24; low query-client. **Фикс:** централизовать ключи в один модуль-фабрику.
3. **Семантика ошибок схлопнута в «нет данных»** — `null` означает и «пусто», и «сбой»; HTTP 200 считается успехом; сетевой сбой кэшируется как «нет текста» на час. → medium #18, #21; low musixmatch, stream.ts:79. И зеркальная крайность — high #10, где ошибка continuation стирает уже загруженное. **Фикс:** различать «пусто»/«ошибка», проверять статус ответа, не ронять загруженный UI из-за сбоя догрузки.
4. **Контракт store→engine через производные строки** — переходы трека передаются строкой videoId и булевым `playing`, а не явными событиями/токенами; часть async-путей без staleness-guard. → high #5, #6; medium #14, #15. **Фикс:** монотонный playToken на переход; guard на устаревшие async-результаты.
5. **Жизненный цикл yt-dlp / гонки downloads-map в Rust** — нет таймаута/kill, ошибка записи оставляет success=true, check-then-insert не атомарен. → medium #2, #3, #4; low #1365. **Фикс:** timeout+kill, `ok=false` на ошибке записи, атомарный get-or-spawn.
6. **Сетевой trust-boundary** — локальный прокси без auth (CSRF/DNS-rebinding), SSRF в cache_cover. → high #1; medium #1. **Фикс:** Host-check + per-launch токен; allowlist хоста + запрет приватных IP/редиректов.
7. **Нет страховки** (мета-причина) — без git/тестов/CI ничему из вышеперечисленного нечем поймать регресс. → high #11.

## Оценка по подсистемам

- Strengths: the stream server correctly binds to the 127.0.0.1 loopback (not 0.0.0.0) on a random port; video_id and cover filename inputs are tightly sanitized (alnum/_/- , length caps, explicit ".." rejection) so path traversal is well-guarded; all external processes (yt-dlp) are spawned with argument arrays and no shell, so there is no command injection; cookies are encrypted with DPAPI on Windows and yt-dlp is deliberately never handed cookies; the Tauri capability http scope is host-limited and the CSP is otherwise restrictive (script-src 'self', object-src/frame-ancestors/form-action 'none'). Weaknesses: the local HTTP server pairs CorsLayer::permissive() with a complete absence of Origin/Host/auth validation on endpoints that spawn processes and write unbounded amounts to disk, making it exposed to CSRF/DNS-rebinding abuse; cache_cover is an unrestricted server-side URL fetch (SSRF primitive); cookie storage silently degrades to plaintext off-Windows; and raw auth cookies are handed to the webview while unsafe-headers is enabled, so any webview compromise yields full Google credentials. Overall the file system / process boundaries are handled carefully, but the network-facing trust boundary (who may talk to the local server, and what URLs it may fetch) is the weak point.

- The Rust backend is well above hobby-project average: thorough doc comments explaining non-obvious decisions (moov-atom streaming rationale, cookie-domain normalization, event dedup), consistent String-error handling with essentially no unwraps on untrusted data, correct input sanitization on both proxy routes (video-id and cover-filename validation rule out traversal), and thoughtful touches like DPAPI cookie encryption, magic-byte MIME sniffing, and the 32 KB stub-download floor. Its weaknesses cluster in concurrency and process lifecycle: check-then-act races on the downloads map and accounts index, no timeout/kill discipline on yt-dlp children, one blocking subprocess call on the main thread, and a completely unauthenticated permissive-CORS localhost server. Nothing here is architecturally unsound — the fixes are all local — but the absence of tests means none of these race/failure paths has ever been exercised deliberately.

- This is a well-crafted subsystem overall: each source is cleanly isolated behind a common Lyrics type, comments explain non-obvious decisions (CORS/User-Agent constraints, LRCLIB get-vs-search race, Musixmatch UpgradeOnly token rejection), the LRC parser handles multi-timestamp lines and 1/2/3-digit fractions correctly, and track-change races are soundly delegated to react-query key identity — I found no data races. The two systemic weaknesses are error handling that collapses every failure into null (defeating retry and caching \"no lyrics\" answers that are really network errors) and trust in remote search relevance without validating the match, which for Genius produces confidently displayed wrong-song lyrics. The time-sync highlight math in lyrics-view is correct at boundaries (linear scan, not binary search, but fine at lyric scale); its real defect is scroll-state lifecycle (no centering on mount, no reset on track change) rather than the sync logic itself.

- Overall this is well-crafted code: comments explain intent and known hazards, there are deliberate race guards (resolve token for stream resolution, StrictMode-safe Tauri listener disposal, persist partialize that never auto-plays on launch), listener/cleanup hygiene is solid, and the floating-window remote-control bridge is a thoughtful design. The main structural weakness is that the store-to-engine contract communicates track transitions through derived strings (videoId) and a boolean `playing` rather than explicit transition events, which is exactly what breaks repeat-one, duplicate-videoId advancement, and the paused-start blip — the store thinks a transition happened while the engine's effects see no dependency change. Async side effects that lack the staleness guards the stream resolver has (auto-radio append, premium detection) form the second cluster of bugs. With no tests, these mode-interaction regressions (shuffle x setQueue, repeat-one x ended) have nothing to catch them.

- Overall this is well-crafted UI code: the async Tauri listen() teardown races are handled consistently with cancelled-flag patterns, the transport/queue snapshot split shows real attention to IPC cost, comments explain non-obvious decisions honestly, and the entity header does composited-property animation correctly. The weak spot is the floating-player state machine: the action-override bridge covers only part of the store surface (menu actions silently break in the floater), and the window-close event conflates user intent with programmatic closes, so layout mode gets corrupted by ordinary queue operations. The dock watcher's timeout-as-drop heuristic and the overlay-zone semantics for floating drags are both approximations that misbehave in reachable states. Smaller polish issues (DnD off-by-one, wheel-unmute, global Tab block) suggest limited testing of secondary interaction paths — unsurprising given there are no tests at all.

- This is an unusually well-crafted persistence layer for an untested codebase: stores are small and single-purpose, persist configs deliberately partialize volatile fields (premium status rederived per launch, playback position never auto-resumes), versioned migrations and a legacy-bucket handoff exist for pinned playlists, selectors use stable EMPTY references, all Tauri listeners are StrictMode-safe, and hydration is race-free because everything uses synchronous localStorage rather than the async tauri-plugin-store. Write amplification is well controlled (search history writes only on submit, sort modes drop default entries, track-source caps its map). The systemic weakness is multi-window awareness: the main and floating windows are independent JS contexts sharing localStorage keys with last-writer-wins semantics, which breaks both the query-cache persister (including a cross-account data resurrection path) and the floatingPinned preference. Account isolation is done by in-session clearing of shared buckets rather than account-scoped keys, which is fragile at shutdown boundaries and lossy for search history; caches with lazy TTL eviction (cover art) also grow without bound.

- Overall this is a carefully written hand-rolled InnerTube layer: parsing is defensive (optional chaining everywhere, tree-walk fallbacks with WeakSet cycle guards, synthesized i.ytimg thumbnails), pagination has real safety rails (token-echo check, 200-page cap, dedupe sets), and the comments documenting YT response-shape quirks are unusually good. The client-init race the module structure suggests is moot because the youtubei.js singleton is never actually used — which is itself the problem: dead dependency plus misleading docs. The main weaknesses are on the write path (mutations trust HTTP 200 and invalidate query keys that don't exist, so optimistic UI can lie) and in copy-paste drift (home/explore extraction helpers duplicated verbatim, readRun vs readRuns already diverged); the heuristic column classifiers and count regexes will keep producing occasional silently-wrong metadata as YT varies locales and layouts.

- This is a well-crafted UI layer overall: the virtualized TrackList handles scrollMargin, measurement, and duplicate-id keys correctly; the Thumbnail fallback chain (iTunes override → disk-cached upgrade → largest API variant, with per-tier onError demotion and cancellation-safe async resolve) is genuinely careful; routes consistently have skeleton/error states and search params are properly validated. The main weaknesses are query-key hygiene drift — three different keys hold liked/playlist track data ("liked-songs", "library/liked-songs-pages", "playlist-pages") with no cross-synchronization, producing the dead invalidation and dead persistence entry found above — and infinite-scroll error handling, which either retries unboundedly or wipes loaded content on a single failed continuation. There are also DOM-side channels (data-videoid querySelector) that were designed before virtualization and now track detached nodes. No tests exist, so none of these regressions would be caught automatically.

- This is an unusually thoughtful hobby codebase: the frontend architecture is genuinely strong, with disciplined layering (raw InnerTube fetch + parsers in lib/innertube, Zustand stores with careful persist/partialize choices, a well-designed cross-window bridge that splits high-frequency transport from low-frequency queue snapshots), pervasive high-quality comments explaining *why* (StrictMode listener races, CORS/CSP workarounds, cache TTL rationale), and consistent per-layer error strategies (best-effort null for lyrics/covers, throw for InnerTube, toast at the UI edge). The weaknesses are all about the absence of guardrails rather than the code itself: no git, no lint, no tests, and no CI means the ad-hoc query-key strings have already drifted (dead invalidations, dead persist whitelist entries), a fully dead youtubei.js client still ships ~270 KB, and verbatim duplications (source-switch flow, infinite-scroll plumbing, time formatters) have started accumulating in the component layer where only convention prevents divergence.
